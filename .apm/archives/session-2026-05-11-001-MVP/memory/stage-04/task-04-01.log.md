# Task 4.1 - Reverie playback at zero + voice disconnect

## Summary

Added `play_reverie_then_disconnect(voice_client) -> bool` to `app/voice.py` and extended `tests/test_voice.py` with four new tests. All 88 tests pass and the full validation pipeline is clean.

## Details

**Dependency integration:** Read `app/voice.py` (existing helpers: `connect`, `play_reverie`, `disconnect`) and `tests/test_voice.py` (existing patterns). Verified that `discord.VoiceClient.loop` is a real attribute (set to `state.loop` in `VoiceClient.__init__`) by inspecting the installed library source — it is present in discord.py 2.7.1, consistent with the Task Prompt.

**Implementation of `play_reverie_then_disconnect`:**
- Creates an `asyncio.Event` (`done`) and a `list[bool]` (`success`) for thread-safe signalling from the `after` callback.
- Captures `voice_client.loop` before calling `play()` so the callback can call `loop.call_soon_threadsafe(done.set)` without accessing the voice client again from the audio thread.
- Wraps `voice_client.play(discord.FFmpegPCMAudio(...), after=after)` in try/except; on synchronous failure sets `success[0] = False` and skips `await done.wait()`.
- If `play()` succeeded, awaits `done.wait()` until the callback fires.
- Always calls `await voice_client.disconnect()` in a try/except; disconnect errors are logged at WARNING and do not affect the return value.
- Returns `success[0]`.

**Test structure:** The four tests use a helper `_make_fake_voice_client()` that wires `fake_client.loop = asyncio.get_event_loop()` so `call_soon_threadsafe` routes correctly without spawning a real thread. Tests that need to drive the `after` callback do so by capturing it in a `side_effect` function, then calling it directly after `await asyncio.sleep(0)` yields control to the coroutine under test.

**Imports added to `app/voice.py`:** `asyncio`, `logging`, `logger = logging.getLogger(__name__)`.
