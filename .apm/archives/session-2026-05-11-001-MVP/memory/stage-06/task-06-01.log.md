## Issues

**PyNaCl CVE-2025-69277 is unresolved.** The fix version (PyNaCl 1.6.2)
is incompatible with discord.py[voice]==2.7.1 which pins PyNaCl<1.6.
Resolving requires a discord.py upgrade. This is a hard constraint —
pip raises ResolutionImpossible if both are pinned simultaneously.

Recommended path: bump discord.py[voice] to the latest release that
ships with PyNaCl>=1.6 support, then re-run pip-audit to confirm the
finding clears. Run the full test suite after the bump. This should be
done before the V1 tag in T6.2.

## Important Findings

**PyNaCl CVE-2025-69277 is blocked by discord.py's own dependency
pin.** discord.py[voice]==2.7.1 requires `PyNaCl<1.6,>=1.5.0`. The
CVE fix is in PyNaCl 1.6.2. These two constraints cannot coexist —
pip refuses to resolve them. The only path to remediation is a
discord.py version bump.

The CVE itself (elliptic curve point validation) is low-risk for
TeaMode's use case — the vulnerability affects atypical custom
cryptography paths, whereas discord.py uses PyNaCl only for standard
voice channel encryption. However, leaving a known CVE open at the V1
tag is not ideal.

**Recommended action for the Manager before T6.2:** check whether a
later discord.py release (2.7.x or 2.8.x) has relaxed or updated the
PyNaCl bound. If so, bump discord.py, re-run pip-audit to confirm
clear, and run `pytest` to verify no regressions. If discord.py has
not yet released with PyNaCl>=1.6 support, document the open finding
explicitly in the release notes and proceed to tag.

**Why PyNaCl must be explicitly pinned if bumped:** as a transitive
dependency it does not appear directly in requirements.txt. If discord.py
is upgraded and its new version loosens the PyNaCl bound to allow 1.6.x,
pip would resolve it freely. An explicit `PyNaCl==1.6.2` line in
requirements.txt would then be needed only if the new discord.py still
allows both 1.5.x and 1.6.x — to lock in the patched version and
prevent accidental downgrades. If the new discord.py forces PyNaCl>=1.6,
no explicit pin is required.
