# Task 3.3 - Active timer message + countdown loop + edit cycle (+ two followup fixes)

## Summary

Implemented the full active-timer surface, then resolved two successive 403
Forbidden errors discovered during smoke testing. The final state routes all
post-modal sends through `interaction.followup.send` and resolves the voice
channel from the click-handler's `interaction.channel` — eliminating the REST
`fetch_channel` call and its permission gate entirely. All 84 tests pass, all
five pipeline checks clean.

## Details

### Original T3.3 implementation (commits 49bd240, 1dc846f, efafb20)

#### `run_countdown` coroutine (`app/session.py`)

Added at module end. Imports: `asyncio`, `time`, `collections.abc.Callable`, `collections.abc.Awaitable`.

- Signature: `async def run_countdown(*, duration_minutes, on_tick, sleep=asyncio.sleep, monotonic=time.monotonic)`.
- Ticks every second from `total_seconds` down to `0` inclusive (61 ticks for a 1-minute session).
- Drift correction: each sleep target is computed from a fixed `start = monotonic()` origin; individual tick latency shifts the next sleep target, not future targets.
- Returns cleanly at zero. Does not call `mark_followup` — caller responsibility.
- No Discord imports; testable in total isolation.

#### `_EditState` dataclass and constants (`app/bot.py`)

Added `_ACTIVE_TIMER_FMT`, `_EDIT_INTERVAL_SECONDS = 10`, `_BACKOFF_FLOOR_DEFAULT = 10.0`, `_BACKOFF_FLOOR_CAP = 60.0`, and `_format_timer` helper.

`_EditState(message, lock, backoff_floor)` — a per-session mutable holder. `TeaModeBot._edit_states: dict[int, _EditState]` initialized in `__init__`.

#### `_on_countdown_tick` method (`TeaModeBot`)

- Skips non-edit-eligible ticks (`seconds_remaining % 10 != 0` and `!= 0`).
- Checks `edit_state.lock.locked()` before acquiring — skips if in-flight.
- On `discord.HTTPException` with `status == 429`: doubles `backoff_floor` up to cap, logs warning.
- On success: decays `backoff_floor` back to `_BACKOFF_FLOOR_DEFAULT`.
- Other HTTP exceptions: logged but not re-raised (timer continues).

#### `IntentionModal.on_submit` — initial implementation

Post-intention sequence as originally implemented:
1. `registry.set_intention(...)` — captures returned `Session` object.
2. `interaction.response.defer(ephemeral=True)`.
3. `interaction.channel.send(_MSG_PARTICIPANT_PROMPT)` — posted via bot HTTP path.
4. `client.fetch_channel(int(session.voice_channel_id))` — validates it is a `VoiceChannel`.
5. `voice.connect(voice_channel)` — on any exception: sends ephemeral `_MSG_VOICE_CONNECT_FAILED` via `interaction.followup.send`, calls `mark_cancelled`, returns early.
6. `registry.mark_active(session_id=...)`.
7. `interaction.channel.send(initial_content)` — posts `🍵 Intention: <text>  ⏳ mm:00` via bot HTTP path.
8. Stashes `_EditState` in `bot._edit_states[session_id]`.
9. `asyncio.create_task(_run_and_followup())` — inner coroutine awaits `run_countdown`, calls `mark_followup`, pops edit state, disconnects voice.

### First smoke test fix — followup.send (commit 588dced)

**Root cause:** `interaction.channel.send(...)` goes through the Discord REST
HTTP path and requires the bot to have explicit View Channel + Send Messages
permission on the voice channel. The welcome embed posted fine because it used
`interaction.response.send_message(...)`, which routes through the interaction
webhook and bypasses that check. The post-modal sends using `channel.send`
produced a `403 Forbidden (50001) Missing Access`.

**Fix in `app/bot.py` — `IntentionModal.on_submit`:**

1. Participant prompt send changed to:
   ```python
   await interaction.followup.send(_MSG_PARTICIPANT_PROMPT, ephemeral=False)
   ```
   The `# type: ignore[union-attr]` comment was dropped because `interaction.followup`
   is non-Optional (unlike `interaction.channel`).

2. Active timer message post changed to:
   ```python
   timer_message = await interaction.followup.send(
       initial_content, ephemeral=False, wait=True
   )
   ```
   `wait=True` ensures the returned `WebhookMessage` (a `discord.Message` subclass)
   has `.edit(...)` and `.id` — the existing edit-cycle code works unchanged.

The voice-connect failure error path was already `followup.send(..., ephemeral=True)`;
no change needed there.

**Test updates in `tests/test_bot_intention.py`:**

- `test_modal_submit_records_intention_and_posts_participant_prompt`: changed
  `inter.channel.send` mock to `inter.followup.send = AsyncMock(side_effect=[None, fake_timer_msg])`;
  updated all assertions to check `followup.send` with `ephemeral=False` and `wait=True`
  on the second call.
- `test_modal_submit_voice_connect_failure_cancels_session`: this test now expects two
  `followup.send` calls (participant prompt with `ephemeral=False`, then error with
  `ephemeral=True`); updated from `assert_called_once_with` to checking the call list.
  Also removed the now-unused `inter.channel = AsyncMock()` line.

### Second smoke test fix — drop fetch_channel (commit b4e0693)

**Root cause:** After the `channel.send` fix, a second `403 Forbidden (50001)
Missing Access` appeared on `client.fetch_channel(int(session.voice_channel_id))`
inside `on_submit`. `fetch_channel` is a fresh REST GET that hits Discord's
per-channel View Channel permission gate independently of the interaction
webhook. The bot's role had View Channel + Connect + Speak + Send Messages on
the voice channel in the facilitator's server — but the code was still
structurally brittle: the voice channel object was already available at the
click site in `_handle_timer_pick` as `interaction.channel` (guaranteed by
the `/teamode` invocation guard). No REST call was needed.

**Fix — option A (thread channel via modal init):**

`IntentionModal.__init__` received a new keyword-only parameter
`voice_channel: discord.VoiceChannel`, stored as `self._voice_channel`.
`on_submit` reads `self._voice_channel` directly. The entire `fetch_channel`
block and its associated try/except for fetch failure and wrong channel type
were removed. The voice-connect failure path (try/except around
`voice.connect`) was retained unchanged.

`_handle_timer_pick` was updated to:
1. Assert `isinstance(interaction.channel, discord.VoiceChannel)` — satisfies
   pyright's narrowing; enforced at runtime by the prior guard.
2. Pass `voice_channel=interaction.channel` to `IntentionModal(...)`.

**Test updates in `tests/test_bot_intention.py`:**

- `_make_component_interaction` helper updated to accept an optional
  `voice_channel: discord.VoiceChannel | None` parameter assigned to
  `inter.channel` — required because `_handle_timer_pick` now asserts the
  channel type.
- `test_facilitator_click_sets_duration_and_opens_modal`: passes
  `MagicMock(spec=discord.VoiceChannel)` as `voice_channel`.
- `test_modal_submit_records_intention_and_posts_participant_prompt`: moves
  fake channel construction above modal init; passes it as `voice_channel=`;
  removes `patch.object(bot.client, "fetch_channel", ...)` and the
  `mock_fetch.assert_called_once_with(444)` assertion.
- `test_modal_submit_voice_connect_failure_cancels_session`: same — passes
  fake channel to modal constructor; removes `fetch_channel` patch.
